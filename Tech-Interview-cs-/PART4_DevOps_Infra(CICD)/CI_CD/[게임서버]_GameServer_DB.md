# 🌐 게임 서버 사용자가 늘때 DB 활용

> 사용자 100명일 때 돌아가던 서버가 이용자수가 늘면 어떻게 처리하느 지

---

## 1. 배경

게임 서버를 운영하고 관리 중인데, 문득 사용자가 갑자기 늘어날 때 서버를 어떻게 활용해야하는 지 궁금했다
DB 쿼리가 느려지고, 응답 지연이 쌓이고, 서버가 다운된다.
대부분 데이터베이스에서 이슈가 생긴다

이번 글에서 \*\*서버-DB 구조 관점에서 사용자 증가에 대응하는 전략을 다뤄보기로 하였다

1. 수직 확장(Scale up)
2. 캐싱(Caching)
3. 샤딩(Sharding)

---

## 2. 수직 확장 (Scaling) 빠르지만 한계

### 개념

서버 머신 자체의 자원을 늘리는 방법이다
CPU 코어 수, 메모리 용량, 디스크 I/O 성능을 관리한다

```
[Before]                    [After]
┌──────────────┐            ┌──────────────────────┐
│ DB Server    │  ───→      │ DB Server (Upgraded) │
│ CPU: 4 core  │            │ CPU: 32 core         │
│ RAM: 16 GB   │            │ RAM: 256 GB          │
│ Disk: HDD    │            │ Disk: NVMe SSD       │
└──────────────┘            └──────────────────────┘
```

### 사용

- 트래픽이 갑자기 늘은 뒤, **빠른 대응이 필요할 때**
- 아직 아키텍처를 바꿀 여유가 없을 때
- 사용자 수가 아직 수만 명 이하 수준일 때

### 장점

- 애플리케이션 코드나 DB 스키마를 건드리지 않아도 된다
- 클라우드 환경(AWS RDS, GCP Cloud SQL 등) 에서는 인스턴스 타입 변경만으로 적용이 가능하다

### 단점

| 문제               | 설명                                          |
| ------------------ | --------------------------------------------- |
| 물리적 한계        | 머신 스펙에는 천장이 존재한다                 |
| 단일 장애점 (SPOF) | 해당 서버가 죽으면 전체 서비스가 영향받는다   |
| 비용 비효율        | 고사양 서버 한 대 가격 >> 저사양 여러 대 합산 |

---

게임 서버의 초기 트래픽 증가에 대해서 수직 확장으로 대응, 데이터베이스 인스턴스의 CPU와 메모리 I/O 성능을 높이는 것이 중요하다

---

## 3. 캐싱 - 데이터베이스 부하를 줄이는 전략

### 캐싱이 필요한 이유

게임 서버에 DB로 들어오는 요청의 대부분은 **같은 데이터를 반복해서 읽는 것**이다.

- 랭킹 조회
- 아이템 목록
- 유저 프로필

데이터를 매번 DB에서 읽으면:

```
클라이언트 -> 서버 -> DB -> 서버 -> 클라이언트 (매 요청마다 반복)
```
