# 🌐 Git cherry-pick으로 선택된 커밋만 개발 브랜치에 적용하기

> 전체 브랜치 머지 없이 특정 커밋만 골라 다른 브랜치에 반영하고 싶을 때 `git cherry-pick`을 활용한다.
> 해당 글에서는 실제 기능 개발 중 겪은 상황을 예시로, cherry-pick의 사용법과 주의 점을 정리하였다.

---

## 1. 배경

기능을 개발하던 중, `feature/example_contents` 브랜치의 **일부 커밋**만 `develop` 브랜치에 먼저 반영해야 하는 상황이 생겼다.
브랜치를 전체를 머지하면 **아직 작성중이거나 잠시 푸쉬한 코드**까지 함께 들어간다는 점이었다.
이때 유효한 해결책이 바로 `git cherry-pick`이다.

> 공식 문서 정의
> `git cherry-pick`은 기존 커밋이 도입한 변경을 현재 브랜치에 하나씩 적용해 **새로운 커밋을 만드는 명령**이다.
> 한 브랜치의 특정 커밋을 골라 다른 브랜치에 적용하는 행위이다.

---

## 2. 이슈 상황

반영이 필요한 커밋은 아래 세게였다.

| 커밋 해시  | 설명                                                                             |
| ---------- | -------------------------------------------------------------------------------- |
| `a0efa0ab` | `[Feature]` 백엔드 시작 관련 프로토콜, 응답, 테스트 코드 추가 및 수정            |
| `a6c36987` | `[Refactor]` 백엔드 시작 핸들러 수정                                             |
| `00542c6e` | `[Refactor]` 백엔드 스타트 프로토콜, 서비스, 테스트 코드 클라 응답값에 맞춰 변경 |

나머지 커밋들은 아직 작업 중인 상태였기 때문에 **전체 브랜치 머지는 어려운**한 이슈였다.

---

## 3. 작업 순서

### 1단계 - 대상 브랜치 체크아웃 및 최신화

`cherry-pick`을 적용할 브랜치로 이동, 원격과 동기화한다.

```bash
git checkout develop
git pull origin develop
```

### 2단계 - 커밋 선택적 적용

`git cherry-pick` 명령에 적용할 커밋 해시를 지정한다.
여러 커밋을 한 줄에 나옇라 수도 있지만, **충돌을 단계적으로 관리**하기 위해 한 커밋씩 적요하는 방법을 선택했다

```
git cherry-pick a0efa0ab
git cherry-pick a6c36987
git cherry-pick 00542c6e
```

> 한 번에 커밋을 나열해 적응하는 것도 가능하나, 추천하지는 않는다.
>
> > ```bash
> > git cherry-pick a0efa0ab a6c36987 00542c6e
> > ```
>
> ```
>
> ```

### 3단계 - 충돌 해결

cherry-pick 도중 충돌이 발생하면 Git이 작업을 중단한다. 아래 순서로 처리한다

```bash
# 1. 충돌 파일을 직접 수정
# 2. 수정된 파일을 스테이징
git add <충돌 파일>

# 3. cherry-pick 재개
git cherry-pick --contiune
```

적용을 취소하고 싶으면 `--abort`를 사용한다

```bash
git cherry-pick --abort
```

### 4단계 - 원격 저장소에 푸시

모든 커밋 적용 및 테스트가 완료되면 원격 브랜치에 반영한다

```bash
git push origin develop
```

## cherry-pick을 선택한 이유

일반적으로 `git merge`를 사용하면 원본 브랜치의 **모든 커밋**이 대상브랜치에 합쳐진다
이번 상황처럼 일부 커밋만 필요할 때 cherry-pick이 더 정확하다

| 상황                                            | 권장 방법                |
| ----------------------------------------------- | ------------------------ |
| 브랜치 전체를 통합할 때                         | `git merge`              |
| 커밋 히스토리를 선형으로 정리할 때              | `git rebase`             |
| **특정 커밋만 선택적으로 적용할 때**            | **`git cherry-pick`** ✅ |
| 긴급 버그 수정을 다른 브랜치에 빠르게 반영할 때 | **`git cherry-pick`** ✅ |

## 주의해야될 부분

cherry-pick은 편하나, **남용하면 히스토리가 복잡해질 수 있습니다.**

- **중복 커밋 발생**: cherry-pick은 동일한 변경을 새로운 커밋으로 복제한다. 즉, 나중에 전체 브랜치를 머지하면 **같은 변경이 두 번 기록**될 수 있다.
- **히스토리 추적 어려움**: 어떤 커밋이 cherry-pick된 것인지 파악하기 어려워져 협업 시 혼란을 줄 수 있다.
- **Atlassian 공식 가이드**에서도 "cherry-pick이 유용하지만 항상 최선의 방법은 아니며, 전통적인 `merge`나 `rebase`가 더 적합한 경우가 많다"고 언급한다.
  > ✅ **권장 사용 시나리오**
  >
  > - 긴급 버그 수정(hotfix)을 여러 브랜치에 동시 적용
  > - 미완성 브랜치에서 완성된 일부 기능만 먼저 반영
  > - 잘못 커밋된 내용을 올바른 브랜치로 이동

## 정리

| 항목         | 내용                                      |
| ------------ | ----------------------------------------- |
| 명령어       | `git cherry-pick <커밋 해시>`             |
| 주요 용도    | 특정 커밋만 선택적으로 다른 브랜치에 적용 |
| 충돌 시 재개 | `git cherry-pick --continue`              |
| 충돌 시 취소 | `git cherry-pick --abort`                 |
| 주의점       | 히스토리 중복, 남용 지양                  |

이번 작업을 통해 미완성 기능을 제외하고 필요한 변경만 `develop`에 안전하게 반영할 수 있었다. cherry-pick은 편리한 명령어지만, **긴급하거나 명확한 이유가 있을 때만** 사용하고 일반적인 흐름에서는 `merge`나 `rebase`를 우선 검토하는 것을 권장한다.
