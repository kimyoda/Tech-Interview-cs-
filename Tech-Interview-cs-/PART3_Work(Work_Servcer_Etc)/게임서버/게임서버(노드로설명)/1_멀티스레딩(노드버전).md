# 1장 멀티스레딩 - 게임서버 개발자가 알아야할 동시성 기초

## 1. 프로그램과 프로세스

**프로그램(Program)**은 디스크에 저장된 실행 파일이다. 이것이 메모리에 올라가 실행되면 **프로세스(Process)**가 된다.
프로세스는 운영체제로부터 독립된 메모리 공간(코드, 데이터, 힙, 스택)을 할당받는다.
프로세스 간에는 기본적으로 메모리를 공유 하지 않기에, 서로 간섭하지 않고 안전하게 실행된다.

```
[프로그램 실행 흐름]

실행 파일(.exe / 바이너리)
        ↓  OS가 메모리에 로드
     프로세스
    ├── 코드 영역  (명령어)
    ├── 데이터 영역 (전역/정적 변수)
    ├── 힙 영역   (동적 할당 메모리)
    └── 스택 영역  (함수 호출, 지역 변수)
```

---

## Node.js에서 프로세스

Node.js에 `child_process` 모듈이나 `cluster` 모듈로 복수의 프로세스를 만들 수 있다.
각 프로세스는 독립된 메모리를 가지며, IPC(프로세스 간 통신)로 데이터를 주고 받는다.

```js
// cluster 모듈로 멀티 프로세스 구성
const cluster = require("cluster");
const os = require("os");

if (cluster.isPrimary) {
  const numCPUs = os.cpus().length;
  console.log(`Primary ${process.pid} 실행 중, CPU: ${numCPUs}개`);

  for (let i = 0; i < numCPUs; i++) {
    cluster.fork(); // 워커 프로세스 생성
  }

  cluster.on("exit", (worker) => {
    console.log(`Worker ${worker.process.pid} 종료, 재시작`);
    cluster.fork();
  });
} else {
  // 워커에서 Express/NestJS에서 실행
  require("./app").listen(3000);
  console.log(`Worker ${process.pid} 시작`);
}
```

## 스레드

**스레드(Thread)**는 프로세스 안에서 실행되는 실행 흐름의 단위이다. 하나의 프로세스는 최소 1개의 스레드(메인 스레드)를 가지며, 여러 스레드를 만들어 동시에 작업을 처리할 수 있다.

프로세스, 스레드 차이
| 구분 | 프로세스 | 스레드 |
|------|----------|--------|
| 메모리 공간 | 독립적 (공유 없음) | 힙/데이터 영역 **공유** |
| 생성 비율 | 높음 | 낮음|
| 통신 방식 | IPC(소켓, 파이프 등) | 공유 메모리에 직접 접근 |
| 오류 격리 | 한 프로세스 오류가 다른 프로세스에 영향이 없음 | 한 스레드 오류가 전체 프로세스에 영향 |
| 문맥 교환 비용| 높음 | 낮음 |

## Node.js에서 스레드 - Worker Threads

Node.js는 기본적으로 **싱글스레드 이벤트 루프**모델이나, `worker_threads` 모듈로 진짜 멀티스레드를 사용할 수 있다.

```js
// main.js
const { Worker, isMainThread, parentPort, workerData } = require('worker_threads');

if (isMainThread) {
  // 메인스레드: 워커 생성
  const worker = new Worker(__filename, {
    workerData: { taskId: data: [1, 2, 3, 4, 5]}
  });

  worker.on('message', (result) => {
    console.log('워커 결과:', result); // 워커 결과
  });

  worker.on('error', (err) => console.log('워커 오류:', err));
  worker.on('exit', (code) => console.log(`워커 종료 코드: ${code}`));
} else {
  // 워커 스레드: 무거운 연산 처리
  const { data } = workerData;
  const sum = data.reduce((acc, n) => acc + n, 0);
  parentPort.postMeessag(sum);
}
```

> 스레드는 같은 프롯세스의 힙 메모리를 공유하기 때문에 빠르나, 그만큼 동기화 문제(레이스 컨디션, 교착상태)에 주의해야 한다

## 멀티스레드 프로그래밍은 언제?

멀티스레드가 항상 답은 아니다. 다음 세가지 상황에서 효과적이라고 생각한다

### 오래 걸리는 업무 1개, 빨리 끝나는 일 여럿을 동시에 처리할 때

게임 서버에서 DB 저장(오래 걸림)을 하며 동시에 네트워크 패킷(빨리 끝남)을 계속 처리해야 할 때

```
싱글스레드
패킷 처리 -> DB저장(대기...) -> 패킷 처리 -> DB저장(대기..) <- 해당 구간 동안 패킷 처리 불가

멀티스레드
스레드1: 패킷처리 -> 패킷처리 -> 패킷처리 (계속 진행)
스레드2:           DB저장 -> DB저장 (별도 처리)
```

### 긴 처리를 진행하는 동안 다른 짧은 일을 처리할 때

게임 서버에서 레벨 데이터 로딩처럼 오래 걸리는 초기화 작업 중에도 로비 채팅은 계속 동작해야 하는 경우다.

### CPU 코어를 모두 활용해야 할 때

CPU 집약적 연산(물리 계산, 경로 탐색, 매치메이킹 알고리즘 등)을 멀티코어로 병렬 처리한다.

```ts
// NestJS: CPU 집약적 작업을 Worker Thread로 분리
import { Injectable } from "@nestjs/common";
import { Worker } from "worker_threads";
import * as path from "path";

@Injectable()
export class PathfindingService {
  findPath(mapData: number[][], start: Point, end: Point): Promise<Point[]> {
    return new Promise((resolve, reject) => {
      const worker = new Worker(
        path.resolve(__dirname, "pathfinding.worker.js"),
        { workerData: { mapData, start, end } },
      );

      worker.once("message", resolve);
      worker.once("error", reject);
    });
  }
}
```

> Node.js경우: I/O작업(DB, 네트워크, 파일)은 비동기 이벤트 루프가 이미 효율적으로 처리한다. Worker Threaeds는 주로 CPU 집약적 연산에 사용한다.

## 4. 스레드 정체

**스레드 정체**는 특정 스레드가 필요한 자원을 계속 얻지 못해 실행되지 못하는 현상이다.

발생원인:

- 우선순위가 높은 스레드가 CPU를 독점
- 특정 뮤텍스를 항상 같은 스레드만 획득
- 스레드 풀이 꽉 찬 상태에서 새 작업이 계속 들어올 때

Node.js에서 이벤트 루프가 블로킹되는 것도 일종의 정체이다. 무거운 동기 연산이 이벤트 루프를 점령하면 다른 요청이 처리되지 못한다.

```js
//  이벤트 루프 블로킹 — 다른 요청이 처리되지 못함
app.get("/bad", (req, res) => {
  // 동기적 무거운 연산으로 이벤트 루프 점령
  const result = heavyCpuTask(); // 수초 소요
  res.json(result);
});

// Worker Thread로 분리 — 이벤트 루프 비워둠
app.get("/good", async (req, res) => {
  const result = await runInWorker(heavyCpuTask);
  res.json(result);
});
```

## 5. 스레드 다룰 때 주의사항

멀티스레드 환경에서 주의해야 하는 것은 **공유 자원에 대한 동시 접근**이다.
두 스레드가 동시에 같은 변수를 읽고 쓰면 \*\*레이스 컨디션(Race Condiktion)이 발생한다.

```
[레이스 컨디션 예시 — 잔액 차감]

초기 잔액: 1000원

스레드A: 잔액 읽음(1000) → 100 차감 계산(900) → ...
스레드B: 잔액 읽음(1000) → 200 차감 계산(800) → ...
스레드A:                                        → 900 저장
스레드B:                                        → 800 저장

최종 잔액: 800원 ← 100원 차감이 사라짐! (올바른 값: 700원)
```

해당 이슈는 잠금또는 다른 연산이 필요하다.

## 6. 임계영역, 뮤텍스

**임계 영역(Critical Section)**은 공유 자원에 접근하는 코드 구간이다.
**뮤텍스(Mutex, mUTUAL eXCLUSION)**는 임계 영역을 보호하는 잠금 장치이다. 스레드가 뮤텍스를 획득하면 다른 스레드는 같은 뮤텍스를 획득하려 할 때 대기한다.

```
뮤텍스 동작 원리
스레드 A: Lock 획득 -> 임계 영역 실행 -> Lock 해제
스레드 B:             (잠금 대기..)
-> Lock 획득 -> 임계 영역 실행 -> Lock 해제
```

Node.js는 싱글스레드 이벤트 루프, 뮤텍스가 필요한 경우가 드무ㅏ, `worker_threads`를 사용할 때나 Redis 등 외부 공유 자원에 접근할 때는 분산 잠금(Distributed Lock)으로 같은 개념을 구현한다.

```js
// Redis를 활용한 분산 잠금 (뮤텍스 역할)
import { Injectable } from '@nestjs/common';
import { InjectRedis } from '@nestjs-modules/ioredis';
import Redis from 'ioredis';

@Injectable()
export class DistributedLockService {
  constructor(@InjectRedis() private readonly redis: Redis) {}

  // 잠금 획득 (SET NX EX: 없을 때만 설정, TTL 지정)
  async acquireLock(key: string, ttlSeconds: number): Promise<boolean> {
    const result = await this.redis.set(
      `lock:${key}`,
      '1',
      'EX', ttlSeconds,
      'NX'  // Not eXists: 키가 없을 때만 설정 (원자적)
    );
    return result === 'OK';
  }

  // 잠금 해제
  async releaseLock(key: string): Promise<void> {
    await this.redis.del(`lock:${key}`);
  }

  // 잠금을 사용하는 래퍼
  async withLock<T>(
    key: string,
    ttlSeconds: number,
    fn: () => Promise<T>
  ): Promise<T> {
    const acquired = await this.acquireLock(key, ttlSeconds);
    if (!acquired) {
      throw new Error(`잠금 획득 실패: ${key}`);
    }
    try {
      return await fn();
    } finally {
      await this.releaseLock(key);
    }
  }
}

// 사용 예시: 아이템 구매 중복 방지
async purchaseItem(userId: number, itemId: number) {
  return this.lockService.withLock(
    `purchase:${userId}`,
    5, // 5초 TTL
    async () => {
      // 임계 영역: 잔액 확인 → 차감 → 아이템 지급
      const balance = await this.getBalance(userId);
      if (balance < item.price) throw new Error('잔액 부족');
      await this.deductBalance(userId, item.price);
      await this.grantItem(userId, itemId);
    }
  );
}
```

## 7. 교착 상태(Deadlock)

**교착 상태(Deadlock)**는 두 개 이상의 스레드가 서로가 가진 자원을 기다리고 영원히 멈추는 상태이다.

```
교착 상태 발생 예시

스레드A: 뮤텍스1 획득 -> 뮤텍스 2 대기...
스레드B: 뮤텍스2 획득 -> 뮤텍스 1 대기...

-> 서로 상대방의 잠금이 풀리길 기다리며 진행 불가
```

교착 상태의 4가지 필요 조건:
| 조건 | 설명 |
|------|------|
| 상호 배제 | 자원을 한 번에 하나의 스레드만 사용 |
| 점유 대기 | 자원을 점유한 채로 다른 자원을 기다림 |
| 비선점 | 다른 스레드의 자원을 강제로 빼앗지 못함 |
| 순환 대기 | 스레드들이 원형으로 자원을 기다림 |

## 8. 잠금 순서의 규칙

교착 상태를 예방하는 가장 실용적인 방법은 모든 스레드가 동일한 순서로 잠금을 획득하는 것이다.

```
교착 상태 위험

스레드A: Lock(뮤텍스1) -> Lock(뮤텍스2)
스레드B: Lock(뮤텍스2) -> Lock(뮤텍스1) <- 순서 불일치

항상 같은 순서
스레드A: Lock(뮤텍스1) -> Lock(뮤텍스2)
스레드B: Lock(뮤텍스1) -> Lock(뮤텍스2) <- 동일한 순서
```

잠금 대상에 고유한 ID나 우선순위를 부여, 항상 낮은 번호 순서(또는 특정 기준) 순서로 획득한다

```js
// 두 유저 간 거래 시 교착 상태 방지
async transferBetweenUsers(fromId: number, toId: number, amount: number) {
  // 항상 userId가 작은 쪽을 먼저 잠금 (순서 고정)
  const [firstId, secondId] = fromId < toId
    ? [fromId, toId]
    : [toId, fromId];

  return this.lockService.withLock(`user:${firstId}`, 5, async () => {
    return this.lockService.withLock(`user:${secondId}`, 5, async () => {
      // 임계 영역: 두 사용자 잔액 동시 처리
      await this.deduct(fromId, amount);
      await this.add(toId, amount);
    });
  });
}
```

## 9. 병렬성과 시리얼 병목(Amdahl's Law)

**암달의 법칙** 병렬화로 얻을 수 있는 향상의 한계를 설명한다.
프로그램의 일부가 반드시 순차적으로 실행되어야 하고, 아무리 스레드를 늘려도 그 비율 이상 빨라지지 않는다.

```
 전체 작업의 20%가 순차 처리 필 수 일 때

스레드 1개: 100% 속도
스레드 2개: 153% 속도
스레드 4개:  250% 속도
스레드 8개:  308% 속도
스레드 ∞개:  최대 500% 속도 (이론상 한계)
```

**시리얼 병목**은 순차 처리 구간이다.

- 공유 자원에 대한 잠금 구간
- 로그 파일 쓰기
- DB 트랜잭션 순차 처리 부분

> 잠금 구간을 최대한 짧게, 범위를 최소화해야 병렬 성능이 높아진다.

## 10. 싱글 스레드, 멀티 스레드 게임 서버

**싱글스레드 게임서버**
모든 처리를 하나의 스레드에서 순서대로 처리한다

장점:

- 구현이 단순, 동기화 문제가 없다
- 디버깅 용이

단점:

- 한 작업이 느리면 전체가 느려진다
- CPU 코어를 하나만 사용한다

Node.js의 이벤트 루프 방식이 대표적이다. I/O는 비동기로 처리, 자바스크립트 실행은 단일 스레드이다.

**멀티스레드 게임서버**
여러 스레드가 병렬로 작업을 처리한다
장점:

- 멀티코어 CPU 활용
- 처리량 향상

단점:

- 동기화 문제(레이스 컨디션, 교착 상태) 발생 가능
- 구현 복잡도 증가
- 디버깅 어려움

```
싱글스레드 Node.js 이벤트 루프 방식

이벤트 루프 (단일 스레드)
  ├── 패킷 수신 → 처리 → 응답
  ├── DB 쿼리 요청 → (비동기 대기) → 결과 처리
  └── 타이머 → 처리

멀티스레드 방식

메인 스레드: 패킷 수신/라우팅
워커 스레드1: 게임 로직 처리
워커 스레드2: 물리 계산
워커 스레드3: DB 처리
```

## 11. 스레드 풀링(Thread Pool)

스레드를 매번 생성/제거하는 것은 비용이 크다. **스레드 풀(Thread Pool)**은 미리 스레드를 만들어놓고 재사용하는 패턴이다.

```
스레드 풀 동작
작업 큐: [작업1][작업2][작업3][작업4]
              ↓       ↓
스레드 풀: [스레드A] [스레드B] [스레드C] (대기 중)

스레드 A <- 작업1 처리
스레드 B <- 작업2 처리
스레드 C <- 대기
(작업 3, 4는 스레드가 빌 때까지 큐에서 대기)
```

Node.js의 `libuv`는 내부적으로 스레드 풀(기본4개)을 사용해 파일 I/O, DNS조회, 암호화 등을 처리한다.

```JS
// NestJS에서 Worker Thread Pool 구현 (piscina 라이브러리 활용)
import Piscina from 'piscina';
import { Injectable, OnModuleInit } from '@nestjs/common';

@Injectable()
```

## 이벤트 (Event)

**이벤트(Event)**는 스레드 간 신호를 주고받는 동기화 수단이다. 특정 조건이 충족되면 대기 중인 스레드를 꺠운다.

```
[이벤트 동작 원리]
스레드A: 작업 완료 -> 이벤트 발신(signal)
스레드B: 이벤트 대기(wait) -> 이벤트 수신 -> 작업 재계
```

Node.js에서 `EventEmitter`가 해당 역할을 한다. 단일 스레드 내에서 비동기 이벤트를 처리하는 방식이다.

```js
// NestJS에서 이벤트 기반 처리
import { Injectable } from '@nestjs/common';
import { EventEmitter2 } from '@nestjs/event-emitter';

@Injectable()
export class GameService {
  constructor(private eventEmitter: EventEmitter2) {}

  async playerLevelUp(playerId: number, newLevel: number) {
    // 레벨업 이벤트 발신 - 여러 구독자가 독립적으로 처리
    this.eventEmitter.emit('player.levelup', { playerId, newLevel});
  }
}

@Injectable()
export class RewardService {
  @OnEvent('player.levelup')
  async handleLevelUp({ playerId, newLevel}) {
    await this.grantLevelUpReward(playerId, newLevel);
  }
}

@Injectable()
export class NotificationService {
  @OnEvent('player.levelup')
  async handleLevelUp({ playerId, newLevel }) {
    await this.sendLevelUpNotification(playerId, newLevel);
  }
}
```

## 13. 세마포어 (Semaphore)

**세마포어(Semaphore)**는 뮤텍스를 일반화한 개념, 동시에 접근 가능한 스레드 수를 제한한다.

- **뮤텍스:** 동시 접근 허용 수 = 1 (0 또는 1)
- **세마포어:** 동시 접근 허용 수 = N (카운터 방식)

```
세마포어 예시 - 동시 DB 연결 3개로 제한

초기값: 3

스레드A 접근 -> 카운터: 2
스레드B 접근 -> 카운터: 1
스레드C 접근 -> 카운터: 0
스레드D 접근 -> 카운터: 0이므로 대기...
스레드A 완료 -> 카운터: 1 -> 스레드D 접근
```

### 세마포어의 또다른 용도: 스레드 간 실행 순서 제어

세마포어는 "스레드 B는 스레드 A가 완료된 후에만 실행"과 같은 순서 제어에도 사용된다. 초기값을 0으로 설정하면 생산자-소비자 패턴처럼 동작한다.

```JS
// NestJS에서 동시 요청 수 제한 (세마포어 역할)
import { Injectable } from '@nestjs/common';

@Injectable()
export class CoucurrencyLimiterService {
  private currentCount = 0;
  private readonly maxConcureent: number;
  private queue: Array<() => void> = [];

  constructor(maxConcurrent = 5) {
    this.maxConcurrent = maxConcurrent;
  }

  async acquire(): Promise<void> {
    if (this.currentCount < this.maxConcurrent) {
      this.currentCount++;
      return;
    }
    // 한계 쵸과 시 큐에 대기
    return new Promise((resolve) => {
      this.queue.push(resolve);
    });
  }

  release(): void {
    this.currentCount--;
    const next = this.queue.shift();
    if (next) {
      this.currentCount++;
      next();
    }
  }

  async run<T>(fn: () => Promise<T>): Promise<T> {
    await this.acquire();
    try {
      return await fn();
    } finally {
      this.release();
    }
  }
}
```

## 14. 원자 조작 (Atomic Operation)

**원자 조작(Atomic Operation)**은 중간에 다른 스레드가 끼어들 수 없는 연산이다.

일반적인 `count++` 연산은 사실 3단계이다.

1. 메모리에서 값 읽기
2. 1 더하기
3. 메모리에 저장

이 사이에 다른 스레드가 끼어들면 레이스 컨디션이 발생한다. 원자 조작은 이 3단계를 하나의 불가분 연산으로 처리한다.

```
원자 조작, 일반조작

일반 count++:
스레드A: 읽기(5) → 계산(6) → ...          → 저장(6)
스레드B:           읽기(5) → 계산(6) → 저장(6)
결과: 6 ← 한 번의 증가가 사라진다.

원자적 incrementAndGet:
스레드A: [읽기-계산-저장](5→6) 원자적으로 처리
스레드B:                        [읽기-계산-저장](6→7) 원자적으로 처리
결과: 7 ← 정상
```

Redis의 `INCR` 명령은 원자적으로 동작, 분산 환경에서 카운터 구현에 적합하다.

```js
// Redis INCR을 활용한 원자적 카운터 (NestJS)
@Injectable()
export class AtomicCounterService {
  constructor(@InjectRedis() private readonly redis: Redis) {}

  // 원자적 증가
  async incrementScore(playerId: number, points: number): Promise<number> {
    const key = `score:${playerId}`;
    return this.redis.incrby(key, points); // incrby는 원자적 연산
  }

  // 원자적 비교 - 설정 (랭킹 갱신)
    async updateRankingIfBetter(playerId: number, score: number): Promise<void> {
    const key = `ranking:${playerId}`;
    // ZADD NX: 없을 때만 추가 (원자적)
    // ZADD GT: 더 큰 값일 때만 갱신 (원자적)
    await this.redis.zadd('leaderboard', 'GT', score, String(playerId));
  }
}
```
