# 포탈 서버란 무엇인가?

---

## 1. 포탈 서버란?

**포탈 서버(Portal Server)**는 플레이어가 구역을 이동할 때 **명시적인 전환 지점(포탈, 관문, 문)**을 통해 서버 간 이동이 이루어지는 구조다.
포탑에 진입, 서버와의 연결을 **완전히 종료**하고 서버에 새롭게 접소갛ㄴ다.
해당 과정에서 짧은 로딩 화면 또는 전환 연출이 등장한다.

일상적인 비유

> 포탈서버는 톨게이트를 통과하며 다음 구간으로 넘어가는 것과 같다. 구간마다 독립적으로 관리되어 운영이 훨씬 단순하다.

---

## 2. 심리스 서버와의 차이

| 구분         | 심리스 서버                  | 포탈 서버                      |
| ------------ | ---------------------------- | ------------------------------ |
| 이동 방식    | 경계를 자연스럽게 통과       | 포탈/관문을 통해 명시적 이동   |
| 전환 시점    | 플레이어가 인식하지 못함     | 플레이어가 능동적으로 선택     |
| 로딩 화면    | 없음                         | 있음 (짧은 전환 화면)          |
| 서버 간 연결 | 경계 영역에서 동시 연결 유지 | 완전히 끊고 새 서버에 접속     |
| 구현 복잡도  | 매우 높음                    | 낮음~중간                      |
| 서버 독립성  | 낮음 (경계 동기화 필요)      | 높음 (완전히 독립)             |
| 대표 게임    | WoW 오픈월드, GTA Online     | 디아블로, 메이플스토리, 리니지 |

> **핵심 철학의 차이**: 심리스는 "이음새를 숨긴다", 포탈은 "이음새를 콘텐츠로 만든다"

## 3. 포탈 서버 작동 흐름

```
[1단계] 플레이어가 포탈에 진입
        │
        ▼
[2단계] 현재 서버 A에서 플레이어 상태 저장
       - 현재 HP / MP / 버프 상태
       - 인벤토리 및 장비
       - 퀘스트 진행 상황
       - 저장 완료 확인 (ACK)
        │
        ▼
[3단계] 서버 A와의 연결 종료
       - 플레이어 세션 정리
       - 서버 A의 플레이어 데이터 메모리에서 제거
        │
        ▼
[4단계] 로딩 화면 / 전환 연출 표시
       - 클라이언트는 다음 서버 B 주소를 World Manager로부터 수신
       - 필요한 리소스 (맵, 에셋) 로드
        │
        ▼
[5단계] 서버 B에서 신규 접속
       - 저장된 상태 데이터를 기반으로 스폰
       - 게임 재개
```

포탈 진입 시 클라이언트 - 서버 통신 흐름

```
[클라이언트]                [서버 A]              [World Manager]          [서버 B]
     │                         │                        │                      │
     │── 포탈 진입 요청 ───────▶│                        │                      │
     │                         │── 목적지 서버 조회 ────▶│                      │
     │                         │◀─ 서버 B 주소 반환 ─────│                      │
     │                         │── 상태 저장 완료 ───────────────────────────▶│
     │◀─ 서버 B 주소 + 토큰 전달│                        │                      │
     │                         │                        │                      │
     │── 연결 종료 ────────────▶│                        │                      │
     │                         │                        │                      │
     │────────────────────────────────────── 서버 B 접속 ──────────────────────▶│
     │◀──────────────────────────────────── 스폰 완료 응답 ─────────────────────│
```

## 4. 서버 구조 설계

각 World Manager만 전체 서버 목록을 관리한다.

```
[ World Manager ]
                     (서버 목록 및 라우팅 관리)
                              │
          ┌───────────────────┼───────────────────┐
          ▼                   ▼                   ▼
  [ 마을 서버 ]        [ 던전 서버 A ]        [ 던전 서버 B ]
  (로비, 상점,          (화염 동굴,            (얼음 미궁,
   퀘스트 NPC)          1~10층)                1~10층)
          │                   │                   │
     포탈 ▶              포탈 ▶               포탈 ▶
  (입장 요청)          (다음 층 이동)         (보스방 진입)
```

**Zone 서버의 역할 분리**
| 서버 유형 | 역할 | 특징 |
|----------|------|------|
| 로비/마을 서버 | 사교, 거래, 퀘스트 수령 | 항상 켜져 있음, 다수 플레이어 |
| 필드 서버 | 오픈 필드 사냥, 이동 | 구역별 독립 운영 |
| 던전 서버 | 인스턴스 던전 | 파티 단위로 독립 생성/삭제 |
| 이벤트 서버 | 기간 한정 콘텐츠 | 이벤트 기간에만 활성화 |

### 전화 시 데이터 처리

- 서버를 넘어갈 때 데이터를 어떻게 안전하게 전달하는 가

**공유 DB 방식**

```
서버 A → DB에 상태 저장 → 서버 B가 DB에서 읽기

장점: 구현 단순, 서버 간 직접 통신 불필요
단점: DB 쓰기/읽기 레이턴시만큼 전환 시간 증가
```

**토큰 전달 방식**

```
서버 A → 암호화된 상태 토큰 생성 → 클라이언트에 전달
클라이언트 → 서버 B 접속 시 토큰 제출 → 서버 B가 복호화하여 상태 복원

장점: DB 없이 빠른 전환 가능
단점: 토큰 크기 제한, 보안 설계 주의 필요
```

**서버 간 직접 전달**

```
서버 A → 서버 B에 직접 플레이어 데이터 전송 → 서버 B 준비 완료 신호
→ 클라이언트에 전환 허가

장점: 빠르고 정확한 전달
단점: 서버 간 통신 구현 필요, 서버 B 주소를 A가 알아야 함
```

**전환 중 예외 처리**

```
[전환 중 연결 끊김 감지]
        │
        ├─▶ DB에 저장 완료된 경우
        │       → 재접속 시 서버 B로 안내, 정상 복원
        │
        └─▶ DB 저장 전에 끊긴 경우
                → 서버 A의 마지막 저장 상태로 롤백
                → 포탈 진입 전 위치로 복귀
```

## 포탈 서버 장단점

**장점**

- **구현 단순성**
  - 서버 간 경계 동기화, 실시간 상태 공유가 필요 없다. 각 서버는 독립적으로 동작한다.
- **운영 유용성**
  - 특정 던전 서버만 점검, 업데이트해도 다른 구역에 영향을 주지 않는다. 새 콘텐츠를 새 서버 인스턴스로 추가하면 된다.
- **확장 용이성**
  - 서버를 추가로 띄워 여러 채널로 운영할 수 있다. 채널 시스템이 포탈 서버 구조에서 자연스럽게 구현된다.
- **비용 효율**
  - 사용하지 않는 던전 서버는 종료, 필요할 떄만 띄우는 온디맨드 방식이 가능하다.

**단점**

- **몰입감 저하**
  - 로딩 화면이 게임 흐름을 끊고 세계가 분리되어 있다는 느낌을 준다.
- **포탈 위치 강제**
  - 지형과 관계없이 포탈이 있어야만 이동이 가능해 설계의 자유도가 줄어든다.
