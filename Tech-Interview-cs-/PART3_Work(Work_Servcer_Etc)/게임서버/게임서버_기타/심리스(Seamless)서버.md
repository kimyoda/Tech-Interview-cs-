## 1. 심리스 서버란?

전통적인 게임 서버는 맵이나 구역이 바뀔 때 "로딩 중..." 화면이 등장한다.
**심리스(Sameless) 서버**는 이 로딩 없이 플레이어가 끊김 없이 이동할 수 있도록 설계된 서버 구조이다.

### 기존 방식, 심리스 방식

| 구분        | 기존 서버 방식                    | 심리스 서버 양식                  |
| ----------- | --------------------------------- | --------------------------------- |
| 구역 이동   | 연결 끊김 -> 로딩 -> 새 서버 접속 | 투명하게 전환, 플레이어 인식 불가 |
| 구조 복잡도 | 단순                              | 복잡                              |
| 몰입감      | 로딩으로 몰입 저하                | 높은 몰입감 유지                  |
| 사례        | 일반 RPG 존 이동                  | WoW, EVE Online, GTA Online       |

> **대표 사례**: World of Warcraft(오픈 월드 대륙), EVE Online(솔라 시스템 전환), 엘든 링(멀티플레이 심리스), GTA Online(자유로운 이동)

## 2. 아키텍처 설계 - 존(Zone)기반 구조

심리시 서버의 핵심은 전체 맵을 여러 **존(Zone)** 으로 나누고, 각 존마다 독립적인 서버 프로세스를 두는 것이다.
존들 사이에 **경계 영역(Border Zone)**을 두어 부드러운 전환을 시도한다.

```
[ Zone A 서버 ] ── [경계 영역] ── [ Zone B 서버 ] ── [경계 영역] ── [ Zone C 서버 ]
     북부 대륙                        중부 평원                        남부 해안
         │                                │                                │
         └────────────────────────────────┴────────────────────────────────┘
                                          │
                                  [ World Manager ]
                             전역 상태 조율 · 플레이어 위치 관리
```

### 주요 구성 요소

- **Zone 서버**: 지정된 맵 구역의 게임 로직을 담당한다. 해당 구역의 플레이어, 몬스터, 오브젝트를 처리한다.
- **경계영역(Border Zone)**: 두 Zone이 겹치는 가상 공간이다. 이 구역에서 플레이어는 두 서버 모두에 동시 등록된다.
- **World Manager**: 전체 존의 상태를 조율하는 중앙 관리자. 어느 서버에 플레이어가 있는 지 추적한다.

### 설계 시 고려사항

- 존의 크기는 서버 한 대가 처리할 수 있는 적정 동시 접속자 수를 기준으로 결정한다.
- 경계 영역의 폭은 네트워크 레이턴시보다 충분히 넓게 서정해야 전환이 자연스럽다.
- World Manager가 단일 장애점이 되지 않도록 이중화 구성이 필요하다.

## 3. 존 간 캐릭터 이동 구현

캐릭터가 존 경계를 넘어가는 순간, 서버 입장에서는 꽤 복잡한 작업이 일어난다.

### 이동 처리 흐름 (핸드오프 프로세스)

1. **경계 영역 진입 감지**
   Zone A 서버가 플레이어의 좌표가 경계 영역에 들어왔음을 감지한다.

2. **Zone B 서버에 사전 등록**
   Zone A가 Zone B에 플레이어 정보(위치, 상태, 인베토리 등)를 미리 전달해 준비시킵니다.

3. **더블 등록 구간(Double Registration)**
   플레이어가 경계 영역 안에 있는 동안, A + B 두 서버 모두 해당 플레이어를 처리한다. 이 구간에서 두 서버는 플레이어 상태를 지속적으로 동기화한다.

4. **권한(Authority) 이전**
   플레이거가 경계를 완전히 넘으면, Zone B가 주 서버(Authority)가 된다. Zone A는 해당 플레이어의 데이터를 정리하고 연결을 해제한다.

5. **클라이언트 재연결(선택)**
   구현 방식에 따라 클라이언트를 Zone B 서버에 직접 재연결, 프록시/게이트웨이를 통해 투명하게 라우팅할 수 있다.
