# 게임서버*API*명세서를*잘쓰는*요령

> API 명세서가 클라와의 계약서라고 생각한다. 대충쓰면 나한테 다시 돌아온다.

---

## 1. 작성계기

게임서버 개발자로 일하다보면 하루에 API를 3~5개 뽑아내야 할 때가 있다.
랭킹, 상점, 이어하기, 이벤트, 시작하기, 아이템 구매, 도전횟수갬 등 기능이 복잡할수록 명세서를 어떻게 쓰느냐가 전체 개발 속도를 좌우한다.
해당 글에서 내가 실제로 사용하는 방식을 예제 기반으로 공유하고자 한다.

---

## 1. API 명세서의 기본구조

명세서는 크게 세 가지 분류이다.

```
1. 개요 (method/ endpoint / 한 줄 설명)
2. 요청 (Request Parameter)
3. 응답 (Response Field)
4. 처리 내용 (서버 로직 흐름)
5. 생플 응답 (실제 테스트 데이터 기반)
```

각 파트를 어떻게 쓴든 지, 실제 예제로 확인한다.

---

## 2. 예제 1 - 랭킹 이벤트 상세 점수 조회 API

| 파라미터명   | 타입 | 배열 여부 | 필수 여부 | 설명           |
| ------------ | ---- | --------- | --------- | -------------- |
| `ranking_id` | uint | false     | true      | 조회할 랭킹 ID |

> 작성팁: `req_is_required` 는 생략하지 않는다.
> 클라쪽과 논의후에 추가하면 된다.

---

### 응답 필드

| 필드명        | 타입                | 배열여부 | 필수여부 | 설명                         |
| ------------- | ------------------- | -------- | -------- | ---------------------------- |
| `detail_list` | 'RankingDetailData` | true     | true     | 일자별 점수 상세 데이터 배열 |

### RankingDetailData

| 필드명           | 타입  | 배열 여부 | 필수 여부 | 설명                             |
| ---------------- | ----- | --------- | --------- | -------------------------------- |
| `record_date`    | date  | false     | true      | 기록 기준 일자                   |
| `best_score`     | float | false     | true      | 해당 플레이 최고 점수            |
| `daily_total`    | float | false     | true      | 해당 일자 전체 점수 합계         |
| `attempt_scores` | float | true      | true      | 시도 횟수별 점수 배열 (최대 3회) |
| `main_card_id`   | uint  | false     | true      | 메인으로 사용한 카드 ID          |
| `sub_card_ids`   | uint  | true      | false     | 서브 카드 ID 배열                |

> **작성 팁**: 중첩 오브젝트는 별도 섹션으로 보기 편하게 분리한다.
> 응답 테이블에 클라쪽에서 이해하기 쉽게 전ㅇ리하여 전달한다.

---

### 처리 내용

```
1. schedule_id 유효성 확인
  - 존재하지 않는 스케줄이면 에러 반호나

2. 해당 스케줄의 이벤트 기간 조회

3. 현재 유저의 완료된 플레이 결과 조회

4. 최종결과가 없으면 에러 없이 `detail_list` = [] 반환

5. 완료된 결과를 플레이 일자 기준으로 정렬 후 상세 데이터 구성
```

> **작성 팁**: "처리 내용"은 번호를 매기고, 조건 분기를 명확하게 들여쓰기로 표현한다.
> 서버 개발자가 로직 설계 순서 그대로 쓰면 클라 개발자도 예외 케이스를 미리 파악할 수 있다.

## 3. 예제 - 미션 보상 수령 API

### 개요

| 항목   | 내용                                |
| ------ | ----------------------------------- |
| Method | POST                                |
| 개요   | 달성된 이벤트 미션의 보상 지급 처리 |
| URL    | `event/claim_mission_reward`        |

---

### 요청 파라미터

| 파라미터명    | 타입 | 배열 여부 | 필수 여부 | 설명                            |
| ------------- | ---- | --------- | --------- | ------------------------------- |
| `shhedule_id` | uint | false     | true      | 보상 수령 대상 이벤트 스케줄 ID |
| `mission_ids` | unit | true      | true      | 보상 수령할 미션 ID 배열        |

---

### 응답 필드

| 필드명             | 타입               | 배열 여부 | 필수 여부 | 설명                                                         |
| ------------------ | ------------------ | --------- | --------- | ------------------------------------------------------------ |
| `missions`         | `EventMissionData` | true      | true      | 갱신된 미션 데이터 배열                                      |
| `moved_to_giftbox` | tinyint            | false     | false     | 소지 상한으로 선물함 이동 발생 여부 (0: 없음 1: 있음)        |
| `current_items`    | `ItemData`         | true      | false     | 갱신된 소지 아이템 배열 (`moved_to_giftbox = 1`일 때만 반환) |
| `updated_cards`    | `UserCardData`     | true      | false     | 갱신된 카드 데이터 배열                                      |

#### EventMissionData

| 필드명     | 타입    | 배열 여부 | 필수 여부 | 설명                                                   |
| ---------- | ------- | --------- | --------- | ------------------------------------------------------ |
| `id`       | uint    | false     | true      | 미션 ID                                                |
| `progress` | uint    | false     | true      | 현재 진행도                                            |
| `status`   | tinyint | false     | true      | 미셩 상태 (0: 미달성 / 1: 달성, 미수령 / 2: 수령 완료) |

---

### 처리 내용

```
1. 요청값 유효성 검사
   - schedule_id 설정 여부 확인
   - mission_ids가 비어 있지 않은 배열인지 확인

2. schedule_id 기준 이벤트 스케줄 조회
   - 보상 수령 가능 기간인지 확인

3. mission_ids 기준 미션 데이터 조회
   - 해당 스케줄에 속한 미션인지 확인
   - status가 1(달성·미수령)인 미션만 처리 가능
   - status != 1인 미션이 포함되면 에러 반환

4. 달성 조건 재확인

5. 보상 목록 구성

6. 보상 지급
   - 소지 상한 초과로 선물함 이동 발생 시 moved_to_giftbox = 1
   - 선물함 이동 발생 시 current_items에 갱신된 아이템 설정

7. 카드 보상 처리
   - 보상에 카드 포함 시 카드 데이터 갱신
   - 갱신된 카드가 없으면 updated_cards = []

8. 미션 상태 갱신
   - 수령 완료된 미션 status → 2

9. 응답 구성
   - missions: 전체 미션 목록 (갱신 반영)
   - moved_to_giftbox, current_items, updated_cards 설정
```

---

### 샘플 응답

> 마찬가지로 **테스트 코드를 실행해 나온 실제 응답값**이다.
> 에러 케이스 샘플(이미 수령한 미션 재요청 등)도 별도로 붙이는 게 좋다.

```json
{
  "response": {
    "missions": [
      {
        "id": 301,
        "progress": 500,
        "status": 2
      },
      {
        "id": 302,
        "progress": 1200,
        "status": 2
      }
    ],
    "moved_to_giftbox": 0,
    "updated_cards": [],
    "current_items": []
  }
}
```

## 4. 명세서 작성 노하우

### 4-1. 샘플 JSOPN은 실제 테스트 데이터로

**✅ Postman / Swagger 실행 결과 복붙**

손으로 작성한 샘플은 반드시 어딘가 틀린다.
타입이 `float`인데 `int`로 쓴다거나, 선택 필드인데 항상 들어오는 것처럼 보이게 쓴다거나.
**테스트 데이터를 그대로 붙이는 게 가장 빠르고 정확하다.**

Postman에서 `Tests` 탭에 아래 코드를 기본으로 넣어둔다.

```javascript
pm.test("Status 200", () => pm.response.to.have.status(200));
pm.test("response exists", () => {
  const json = pm.response.json();
  pm.expect(json).to.have.property("response");
});
```

---

### 4-2. Swagger 공유, Postman 테스트

```
1. 명세서 초안 작성 (마크다운)
2. Swagger(OpenAPI) YAML 변환 → 클라에 공유
3. Postman Collection 만들어서 팀 내부 테스트
4. 테스트 통과 결과 → 샘플 JSON 업데이트
```

Swagger는 클라이언트가 **스펙을 눈으로 확인**하기 좋고,
Postman은 **실제 동작을 직접 찔러보기** 좋다.
둘 다 쓰면 된다. 굳이 하나만 고를 필요 없다.

### 4-3. 클라이언트 개발자와 소통하는 방법

명세서를 올린다고 끝이 아니다.
클라 개발자가 가장 많이 물어보는 건 이거다.

| 질문 유형                        | 예방 방법                                  |
| -------------------------------- | ------------------------------------------ |
| "이 필드 항상 오나요?"           | `res_is_required` 명확히 표시              |
| "이 값 null이 올 수도 있나요?"   | nullable 명시 또는 샘플에 null 케이스 포함 |
| "배열이 비어있을 수 있나요?"     | 처리 내용에 빈 배열 반환 조건 명시         |
| "이 타입 int인가요 float인가요?" | 샘플 JSON을 실제 테스트 데이터로 작성      |

**"처리 내용"에 에러 반환 조건을 빠짐없이 적는 게 핵심이다.**
클라가 예외 처리를 어떻게 할지 결정하려면 어떤 케이스에서 에러가 오는지 알아야 한다.

### 4-4. 중첩 오브젝트는 링크나 별도 섹션으로

**✅ 별도 섹션 or 공통 오브젝트 문서**

공통으로 자주 쓰는 `ItemData`, `UserCardData` 같은 타입은
`00_common_object.md` 같은 파일로 분리해서 링크 걸어두면 좋다.
여러 API에서 반복해서 쓰는 구조는 한 군데서 관리해야 나중에 수정이 편하다.

## 마치며

명세서를 잘 쓰는 게 결국 **내 시간을 아끼는 일**이다.
클라 개발자가 물어볼 질문을 미리 차단하고,
테스트 단계에서 나올 오해를 미리 막는다.
