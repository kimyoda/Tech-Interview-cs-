# 게임서버*API*명세서를*잘쓰는*요령

> API 명세서가 클라와의 계약서라고 생각한다. 대충쓰면 나한테 다시 돌아온다.

---

## 1. 작성계기

게임서버 개발자로 일하다보면 하루에 API를 3~5개 뽑아내야 할 때가 있다.
랭킹, 상점, 이어하기, 이벤트, 시작하기, 아이템 구매, 도전횟수갬 등 기능이 복잡할수록 명세서를 어떻게 쓰느냐가 전체 개발 속도를 좌우한다.
해당 글에서 내가 실제로 사용하는 방식을 예제 기반으로 공유하고자 한다.

---

## 1. API 명세서의 기본구조

명세서는 크게 세 가지 분류이다.

```
1. 개요 (method/ endpoint / 한 줄 설명)
2. 요청 (Request Parameter)
3. 응답 (Response Field)
4. 처리 내용 (서버 로직 흐름)
5. 생플 응답 (실제 테스트 데이터 기반)
```

각 파트를 어떻게 쓴든 지, 실제 예제로 확인한다.

---

## 2. 예제 1 - 랭킹 이벤트 상세 점수 조회 API

| 파라미터명   | 타입 | 배열 여부 | 필수 여부 | 설명           |
| ------------ | ---- | --------- | --------- | -------------- |
| `ranking_id` | uint | false     | true      | 조회할 랭킹 ID |

> 작성팁: `req_is_required` 는 생략하지 않는다.
> 클라쪽과 논의후에 추가하면 된다.

---

### 응답 필드

| 필드명        | 타입                | 배열여부 | 필수여부 | 설명                         |
| ------------- | ------------------- | -------- | -------- | ---------------------------- |
| `detail_list` | 'RankingDetailData` | true     | true     | 일자별 점수 상세 데이터 배열 |

### RankingDetailData

| 필드명           | 타입  | 배열 여부 | 필수 여부 | 설명                             |
| ---------------- | ----- | --------- | --------- | -------------------------------- |
| `record_date`    | date  | false     | true      | 기록 기준 일자                   |
| `best_score`     | float | false     | true      | 해당 플레이 최고 점수            |
| `daily_total`    | float | false     | true      | 해당 일자 전체 점수 합계         |
| `attempt_scores` | float | true      | true      | 시도 횟수별 점수 배열 (최대 3회) |
| `main_card_id`   | uint  | false     | true      | 메인으로 사용한 카드 ID          |
| `sub_card_ids`   | uint  | true      | false     | 서브 카드 ID 배열                |

> **작성 팁**: 중첩 오브젝트는 별도 섹션으로 보기 편하게 분리한다.
> 응답 테이블에 클라쪽에서 이해하기 쉽게 전ㅇ리하여 전달한다.

---

### 처리 내용

```
1. schedule_id 유효성 확인
  - 존재하지 않는 스케줄이면 에러 반호나

2. 해당 스케줄의 이벤트 기간 조회

3. 현재 유저의 완료된 플레이 결과 조회

4. 최종결과가 없으면 에러 없이 `detail_list` = [] 반환

5. 완료된 결과를 플레이 일자 기준으로 정렬 후 상세 데이터 구성
```

> **작성 팁**: "처리 내용"은 번호를 매기고, 조건 분기를 명확하게 들여쓰기로 표현한다.
> 서버 개발자가 로직 설계 순서 그대로 쓰면 클라 개발자도 예외 케이스를 미리 파악할 수 있다.
