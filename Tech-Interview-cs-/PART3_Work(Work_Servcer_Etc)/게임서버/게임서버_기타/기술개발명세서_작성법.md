# 기술개발명세서를 작성하는 법 팁, 설계문서

> 기획서를 받고 바로 코딩 들어가기 전, 기술개발 명세서를 쓰며 일정 및 설계구조를 생각한다.

---

## 작성계기

개발 쪽에서 기술개발명세서라고 부르는 문서가 있다.
팀마다 부르는 명칭은 다를 수 있다. 테크스펙, 설계문서, 개발 스펙 등등..
-> 해당 기능을 어떻게 만들 건지 개발자가 기획서를 보고 정리한 문서
라고 나는 생각한다.

API 명세서는 문서의 일부분이다. 실제 현장에서 기술개발명세서는 개발에 도움을 주기 위해 작성하였다.

- 플로우 설계(API 호출 순서와 상태전이)
- 핸들러별 처리 로직
- DB테이블 구조와 역할 정의
- 운영툴 / 관리자 기능 설계
- 기확쪽과 논의 내용 및 정리된 구조설계
- 미결사항 (추후 확인)
- 일정 산출 근거

나는 이렇게 개발명세서를 작성해도 위와같이 못한적도 있으나, 일정 산출, 그리고 대략적인 기능들, 구조를 잡을 때 꽤나 큰 도움이 되었기에 해당 작성 팁을 공유하고자 한다.

---

## 1. 기술개발명세서 작성

작년에 처음 일정을 맡았을 때, 홈배너에 관해 설계 및 추가하는 작업이었는데 첫 작업이라 어떻게 할 지 막막했을 때, 그때 옆팀 서버 과장님과 주임님이 기술개발명세서를 한번 작성해보라고 하셨다.
그 이후 기획서를 보고 작성한 기술개발명세서를 보시고 연결하는 디비가 `master` 인지 `manage`인지 운영툴에서 입력하여 적용인 지 등 조언을 해주셨고 그에 따라 개발명세서를 수정하고 일정을 산출하며 설계에 관해 고민을 하였다.

```
기획서 초안
→ 기획 1차 문의 (모호한 부분 정리)
→ 기술개발명세서 초안 작성
→ 기획 확인 (논의 내용 반영)
→ 클라이언트 개발자와 API 흐름 합의
→ 명세서 확정
→ 개발 시작
```

개인적으론 위의 명세서를 클라이언트 쪽과 얘기를 했어야 했는데, 그러질 못한 부분이 아쉬워 다음에 신규업무를 맡으며 적응을 할 고민도 있다.

## 2. 명세서 구조 - 무엇을 넣어야 하는 지

실제로 작성했었던 개발명세서 구조인데, 물론 사람에 따라 다를수도 있을꺼라 생각한다. 참고많 ㅏ시길!

```
1. 개요
    - 작성 기준 날짜
    - 대상 기능 / 게임 모드
    - 작성자

2. 1차 문의 내용
    - 기획서에서 불명확한 부분
    - 기획에 전달한 질문 목록
    - 확인된 답변

3. API 구성 (핸들러 목록)
    - API URL / Handler 이름 / 설명 / 상태(신규/수정/유지)

4. 핸들러별 상세 명세
    - 요청 파라미터
    - 응답 필드
    - 처리 내용 (로직 흐름)
    - 응답 예시 (테스트 데이터)

5. 기획 논의 내용 정리
    - 기획쪽과 결론 된 내용
    - 미결 혹은 정해지지 않은 부분

6. 운영툴 / MGMT 설계(데이터)
    - Master 데이터 관리
    - 운영툴 필요 기능, 유저별 저장 데이터 구조

7. DB 테이블 구조
    - 관련 테이블 목록
    - 각 테이블 역할, 주요 컬럼

8. 일정 산출
```

페이지는 생각보다 적을 수도 있으나, 어떤 핸들러를 어떤 명칭으로, 컬럼의 경우 어떻게 디비 구조 및 설계를 할지 생각하는데 시간이 꽤 걸렸다.
필요한 정보를 작성하는게 목적이다.

## 3. 1차 문의 내용

기획서를 숙지하고 문서화한다
작성 형식은 이와 같다.

### 1차 문의

#### 확인 필요 사항

1. 강제 종료 시 이미 소모한 아이템은 복구하는가?
   → 기획 답변: 복구하지 않음. 강제 종료되어도 아이템은 차감 유지.

2. 연습 모드에서 아이템을 소모하는가?
   → 기획 답변: 추후 논의 필요. 현재는 소모하지 않는 방향으로 임시 적용.

3. 글로벌 미션 달성 보상은 유저가 직접 버튼을 눌러 수령하는가?
   → 기획 답변: 버튼 수령 방식 아님.
   참여 유저가 게임 진입 시 자동으로 선물함에 지급.

추후에 기획쪽에서 논의할 때 별도로 상황을 정리할 수 있다

> 기획 쪽과 문의하고 대화한 내용은 문서에 남기는 것이 필요하다

### 3-2. API 구성(핸들러 목록)

기능에 필요한 API를 한 눈에 볼 수 있는 테이블이다
상태 컬럼이 중요하다.

```
## API 구성

| API URL                        | Handler                    | 설명                           | 상태       |
|-------------------------------|----------------------------|--------------------------------|------------|
| `stage/overview`              | OverviewHandler            | 진입 시 전체 상태 조회         | 수정       |
| `stage/launch`                | LaunchHandler              | 라운드 시작, 세션 생성, 자원 차감 | 유지     |
| `stage/commit_round`          | CommitRoundHandler         | 라운드별 결과 저장             | **신규**   |
| `stage/finalize`              | FinalizeHandler            | 저장된 결과 기준 최종 정산     | 수정       |
| `stage/leaderboard`           | LeaderboardHandler         | 순위 조회                      | 유지       |
| `stage/purchase_ticket`       | PurchaseTicketHandler      | 추가 참여권 구매               | 추후 확인  |
```

### 3-3. 핸드럴별 처리 내용 - 로직 흐름을 글로 써본다

API 명세서에서 요청/응답 테이블도 중요하나 어떻게 처리할 것인지 처리 내용도 필요하다.
코드를 쓰기 전에 로직을 글로 써보면 생각하는 것도 중요하다.

```
### CommitRoundHandler — 처리 내용

1. `session_token` 기준으로 진행 중인 세션(status = ACTIVE)을 확인한다.
   - 세션이 없으면 에러를 반환한다.

2. 요청한 `round_seq`가 현재 다음 순서인지 확인한다.
   - `last_committed_seq + 1`과 일치하지 않으면 에러를 반환한다.

3. 동일 `session_token` + `round_seq` 조합의 중복 저장을 방지한다.
   - 이미 저장된 기록이 있으면 에러를 반환한다. (멱등성 보장)

4. `round_result_logs`에 해당 라운드의 결과를 저장한다.

5. `player_sessions.last_committed_seq`를 갱신한다.
   - `total_points += points`

6. 시즌 누적 기록 대상으로 등록한다.

7. 최대 라운드 수(3회)에 도달했으면 `is_finished = true`로 설정한다.

8. COMMIT_ROUND 로그를 저장한다.

9. 현재 진행 상태를 응답으로 반환한다.
```

> 구현 전에 로직 구멍이나 오류를 발견하여 실제 코드를 짤 떄 실수나 시간을 줄일 수 있다. 처리 내용과 실제 코드를 비교하며 케이스를 좀 더 정리할 수 있다. 번호를 매기고 들여쓰기로 표현하는데, 다만 실제 코드를 작성하면서 달라진 설계/구조에 따라 로직도 바뀔수도 있으니 주의할 것

### 3-4. 운영툴(관리자 기능) / MGMT(디비 구조) 설계

게임 서버 개발에서 운영툴(관리자)설계를 나중으로 미루는 것이 아닌 초기부터 세팅을 하고 그 다음 클라와 기획에 맞춰서 정리하는 것이 중요하다
나중에 추가하려면 조금 귀찮긴하나, 미리 어느정도 설계를 잡은 뒤 정리를 하면 된다.

```
## 운영툴 / MGMT 설계

### Master vs 운영툴 역할 구분

| 구분 | 역할 | 예시 테이블 |
|------|------|-------------|
| Master | 기준값 / 정의 / 스케줄 관리 | `season_schedules`, `stage_constants`, `reward_groups` |
| 운영툴 (MGMT) | 시즌별 실제 데이터 지정 | Boost Card List, Blocked Card List |
| User/Log | 유저 진행 상태 / 결과 / 로그 조회 | `player_sessions`, `stage_action_logs` |

### 운영툴 필요 기능 목록

| 기능 | 설명 | 기준 테이블 |
|------|------|-------------|
| 시즌 스케줄 관리 | 기간, 부스트 타임, 보상 그룹 설정 | `master.season_schedules` |
| 스테이지 상수 관리 | 배율, 쿨다운, 참여 횟수 설정 | `master.stage_constants` |
| 부스트 카드 지정 | 시즌별 부스트 카드 지정 | 운영툴 신규 추가 |
| 사용 불가 카드 지정 | 시즌별 제한 카드/캐릭터 지정 | 운영툴 신규 추가 |
| 순위 조회 | 누적/일차/확정 순위 조회 | `ranking.season_leaderboard` |
| 유저 진행 상태 조회 | 현재 라운드/포인트/세션 확인 | `user.player_sessions` |
| 로그 조회 | OVERVIEW/LAUNCH/COMMIT/FINALIZE 로그 조회 | `log.stage_action_logs` |
```

### 3-5. DB 테이블 구조 - 테이블 역할, 관계 명시

이런 서버 작업에서는 DB구조 및 설계가 더 중요하다고 생각한다.
기능이 복잡하고 저장되는 테이블들이 많을 수록 DB 설계가 잘못되면 나중에 고치는게 많이 힘들다.
명세서에 테이블의 역할, 핵심컬럼, 테이블 간 관계를 간단하게라도 명시하는게 도움이 된다.

```
## DB 테이블 구조

### player_sessions (유저 세션 진행 상태)

역할: 현재 진행 중인 라운드 세션을 관리한다. 한 유저당 1개의 활성 세션을 가진다.

| 컬럼 | 타입 | 설명 |
|------|------|------|
| `user_id` | bigint | 유저 ID |
| `season_schedule_id` | int | 시즌 스케줄 ID |
| `session_token` | varchar | 세션 토큰 (시작 시 생성) |
| `status` | enum | IDLE / ACTIVE |
| `last_committed_seq` | tinyint | 현재까지 완료한 라운드 순번 |
| `total_points` | float | 라운드별 포인트 누적합 |
| `is_boost_active` | tinyint | Launch 시점의 부스트 타임 여부 |
| `point_multiplier` | float | Launch 시점의 배율 |
| `main_card_id` | int | 메인 카드 ID |
| `sub_card_ids` | json | 서브 카드 ID 배열 |
| `consumed_item_ids` | json | 사용 아이템 ID 배열 |
| `last_active_at` | datetime | 마지막 활동 시각 |

---

### round_result_logs (라운드별 결과)

역할: CommitRound 시점에 각 라운드의 결과를 저장한다.

| 컬럼 | 타입 | 설명 |
|------|------|------|
| `session_token` | varchar | 세션 토큰 |
| `round_seq` | tinyint | 라운드 순번 (1~3) |
| `points` | float | 해당 라운드 포인트 |
| `is_penalty` | tinyint | 패널티 여부 |
| `round_detail` | json | 상세 결과 데이터 |
| `created_at` | datetime | 저장 시각 |

**비고**: `session_token + round_seq` 복합 유니크 인덱스로 중복 저장을 방지한다.

---

### 테이블 관계 요약
season_schedules (1)
└─ player_sessions (N) : season_schedule_id 기준
└─ round_result_logs (N) : session_token 기준
└─ final_stage_results (1) : Finalize 최종 결과
```

## 5. 클라이언트와 개발자와 협업

명세서가 완성되면 관련 데이터를 클라 개발자에게 공유해야하는데, 우리팀의 경우에는 개발명세서가 아닌 API 마크다운 명세서로 작성하여 API로 소통하여 해당 기술개발명세서를 기반으로

API 마크다운 명세서를 만들어서 공유한 뒤 응답값 및 요청 데이터를 조율하여 진행한다.

---

실전에서 느꼈던 점들

명세서는 짧아도 된다. 단, 빠지면 안 되는 게 있다.

처리 내용의 에러 케이스
기획 논의 결론
미결 사항 명시
DB 테이블 역할

응답 예시는 실제 테스트 데이터로 만든다.
손으로 쓴 JSON은 반드시 어딘가 틀린다.
Postman으로 실제 호출해서 나온 응답을 그대로 붙인다.

미결 사항을 "추후 확인"으로 넘기되, 반드시 추적한다.
체크박스로 관리하고, 개발 시작 전에 전부 닫혀 있어야 한다.
열린 채로 개발 시작하면 중간에 변경이 생긴다.

일정 산출에 리스크 항목을 명시한다.
불확실한 부분은 산출에서 제외하고, 제외했다는 걸 문서에 적는다.
그래야 일정이 바뀌었을 때 이유를 설명할 수 있다.
