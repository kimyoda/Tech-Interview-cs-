# 배치로 글로벌 누계 점수를 집계할 때 고민했던 것들

이벤트 기능을 개발하다가 "전체 유저가 달성한 점수의 합계"라는 작업을 진행하게 되었다.

> 이벤트 기간 동안 전체 유저의 누계 점수가 100,000점을 돌파하면 모든 참여자에게 보상 지급

접수를 합산해서 Redis로 집계해야 하나? 생각했는데, 해당 점수를 실시간은 힘들고 1시간이나 30분, 10분, 5분 단위로 갱신을 해야한다는 점을 어떻게 해야할 지 고민이 되었다.

-> 글로벌 누계 점수를 설계, 배치로 집계, 실제로 잘 돌아가는지 검증한 과정을 정리하였다.

---

## 1. 뭘 저장해야하는 지 정의

개인 랭킹이랑 비슷하게 처리하면 되지 않나 생각했다.
기획서를 다시 보니, 목적이 다르다고 느꼈다.

| 구분        | 목적                               | 필요한 값            |
| ----------- | ---------------------------------- | -------------------- |
| 개인 랭킹   | 유저별 순위 조회                   | 유저 ID + 점수       |
| 글로벌 미션 | 전체 유저 점수 합산 조건 달성 여부 | **단일 숫자 (총합)** |

글로벌 미션은 몇 점인지는 관심이 없다.
전체 점수의 합이 조건을 달성했는 지 판단해야 한다.

Redis Sorted Set같은 복잡한 구조가 필요 없다고 생각했다.
이에 Redis String 하나에 총합 숫자 하나만 넣으면 될꺼라고 느꼈다.

---

## 2. 어떤 점수를 합산해야 하는 지?

글로벌 미션 집계에서 연습 플레이 점수 제외, 정식 플레이 점수만 합산해야 한다.
어떤 플레이 유형을 포합할 것인지 확인.

---

## 3. 배치 커맨드로 주기적으로 재집계한다

누계 점수를 갱신하는 방법은 두 가지라고 생각했다.

A. 플레이가 끝날 때마다 Redis 값을 직접 증가
B. 주기적으로 DB를 집계해서 Redis에 덮어 씌우기
C. 아예 Socket서버를 통해 실시간으로 반영
이 중에서 C는 사실상 너무 과하다는 의견과 채팅 서버같이 실시간 소통은 필요 없을꺼라 얘기하여 두 가지 방법중에 생각했다.

|             | 방법 A (실시간 증가)      | 방법 B (배치 재집계)  |
| ----------- | ------------------------- | --------------------- |
| 반영 속도   | 즉시                      | 최대 N분 지연         |
| 정합성 위험 | 중복 반영, 롤백 처리 복잡 | DB 기준으로 항상 정확 |
| 구현 복잡도 | 높음                      | 낮음                  |

이번 케이스에서는 약간의 반영 지연이 허용되는 기능 이었기 때문에 방법 B를 선택했다.

배치 커맨드를 5분 주기로 실행해서 DB에서 직접 합산한 값을 Redis에 덮어씌운다.
이 방식은 Redis 값이 날아가도 다음 집계 시점에 자동으로 복구 된다는 장점도 있다.

---

## 4. 배치 커맨드는 스케줄러에 등록해 자동으로 실행된다

집계 커맨드는 직접 실행하는게 아닌 **스케줄러에 등록해 주기적으로 자동 실행**하는 구조다.
내가 세팅한 스케줄러는 5분마다 커맨드가 돌면서 DB에서 누계점수를 집계하고 Redis에 덮어씌우는 것이다.
플레이 결과가 발생하자마자 반영되는건 WebSocket같은 실시간 서버를 사용해야 하므로 해당 건은 반려하였고, **스케줄러 실행 주기에 따라 최대 5분 지연**이 생긴다.

---

## 5. `$this->info()` 는 로그가 아니다

배치를 처음 만들고 배치 서버를 패치하여 확인하고자 하였는데 로그가 남아있지 않았다. (기능은 잘 작동하였다)
커맨드 안에서 사용하는 `$this->info()`는 **콘솔 출력**이다.
`storage/logs` 같은 파일로그에는 남ㅈ지 않는다.

| 코드             | 콘솔 출력 | 로그 파일 |
| ---------------- | --------- | --------- |
| `$this->info()`  | O         | X         |
| `$this->error()` | O         | X         |
| `Log::info()`    | X         | O         |
| `Log::error()`   | X         | O         |

ㅅ스케줄러로 자동 실행되는 배치면 운영 중에 터미널을 볼 수가 없다. **성공/실패 여부를 파일 로그로 남기려면 `Log::info`, `Log:error()` 를 써야한다**
다만 5분마다 실행되는 배치면 하루에 288번 로그가 쌓이기에 확인이 되었으면 해당 로그는 지우는게 나중에 다른 부분을 확인할 떄 편하다고 생각한다.

---

## 6. 배치 컨테이너에서 로그 설정을 확인한다

로그가 남지 않는다고 느낄 때 로그 설정을 확인하였다.
배치가 도는 컨테이너에 접속해서 `.env`의 `LOG_LEVEL`값을 확인하였다.

```
LOG_CHANNEL=stack
LOG_CHECK = debug
```

해당 설정이면 `Log::info` 이상은 파일에 남는다.
설정이 맞는데도 로그가 안보이면, 호출 자체가 없는지 확인하자.

---

## 7. Redis 값을 직접 확인

코드를 다 짜고 커맨드를 실행했는데 동작하지 않길래, **Redis에 실제로 값이 들어갔는지 직접 확인**하였다.

Redis CLI에 접속해서 키를 조회하였다.

```bash
redis-cil
keys 'global_mission'
```

해당 건을 하면서 Laravl의 Redis 설정에 prefix가 있으면, 실제 저장도니 키 으림에 prefix가 자동으로 붙는다.

예를들어 코드에서 `event:global_mission:202:total_score`로 저장했어도
Redis CLI에서 `myapp_event:global_mission:202:total_score`처럼 보일 수 있다.
코드에서는 prefix 없이 사용햐도 괜찮으나, **CLI에서 직접 조회할 때는 prefix를 함께 입력해야 한다.**

```
get myapp_event:global_mission:202:total_score
```

이때 숫자가 제대로 확인할 수 있으면 배치가 정상적으로 작동하고 있다는 것이다.

---

## 8. 테스트는 출력이 아닌 Redis 값을 검증한다

배치 커맨드 테스를 통해 해당 배치가 동작할 수 있는지 체크하였다.
해당 기능은 Redis 캐시가 올바르게 갱신되었는가를 확인해야 한다.
캐시 서비스를 직접 호출해서 Redis에 저장된 값을 단언한다.
"점수가 올바르게 집계된다"는 행위 자체를 검증하였다.

테스트에서 특히 챙겨야 할 케이스는 다음과 같다.

- 정식 플레이 점수만 합산되고, 연습 플레이는 제외되는가
- 여러 유저의 점수가 모두 합산되는가
- 진행 중인 스케줄만 집계되고, 종료된 스케줄은 제외되는가
- 배치를 두 번 실행했을 때 값이 누적이 아니라 덮어쓰기 되는가

---

## 9. Redis 캐시 방식의 위험 요소를 인지하고 있어야 한다

Redis는 원본 저장소가 아니라 캐시다. 아래 상황들을 미리 인지하고 있어야 한다.
Redis 키가 날아가면 다음 집계 전까지 값이 0으로 조회된다
원본 데이터는 DB에 있으므로 다음 배치 실행 시점에 자동 복구된다.
그 사이 잠깐 0으로 보일 수 있다는 점만 기획팀과 공유해 두면 된다.

스케줄러가 멈추면 값이 갱신되지 않는다
배치 의존 구조이므로 스케줄러 모니터링은 필수다.

플레이 직후 즉시 반영되지 않는다
5분 주기라면 최대 5분 지연이 발생한다.
이 지연이 허용되는 기능인지 기획 단계에서 명확히 확인해야 한다.

---

## 10. 제 이슈: Redis를 확인하는 방향이 달랐다

마지막으로 실제 배치가 정상적으로 동작하는지 확인하는 과정에서 중요한 이슈가 있었다.

처음에는 배치가 실행되지 않거나 Redis에 값이 저장되지 않는다고 생각했다.
하지만 확인해보니 문제는 배치 로직 자체가 아니라, **내가 Redis 값을 확인하던 방향이 달랐던 것**이었다.

이번 글로벌 미션 누계 점수는 일반 Redis가 아니라 `ranking` Redis 연결을 사용하도록 구현되어 있었다.

즉, 코드에서는 아래처럼 Redis에 값을 저장하고 있었다.

```php
Redis::connection('ranking')->set($key, $totalScore);
```

그런데 확인할 때는 기본 Redis 쪽을 보고 있었기 때문에 값이 없는 것처럼 보였다. 또한 Laravel Redis 설정에는 prefix가 적용되어 있어서, 코드에서 사용하는 키와 Redis CLI에서 실제로 보이는 키 이름이 달랐다.

예를 들어 코드에서는 아래 키를 사용한다.

```php
event:global_mission:202:total_score
```

하지만 Redis CLI에서 실제로 확인해야 하는 키는 prefix가 붙은 형태였다.

```bash
jlpga_database_event:global_mission:202:total_score
```

그래서 Redis CLI에서 확인할 때는 ranking Redis 쪽으로 접속하거나, 해당 Redis 연결이 바라보는 host/port/db를 기준으로 확인해야 했다.

```bash
redis-cli
keys '*global_mission*'
get jlpga_database_mini:global_mission:202:total_score
```

이 부분을 확인하고 나서야 배치가 정상적으로 Redis에 값을 저장하고 있다는 것을 확인할 수 있었다.
즉, 실제 이슈는 다음과 같았다.

| 구분             | 내용                                                                             |
| ---------------- | -------------------------------------------------------------------------------- |
| 처음 의심한 부분 | 스케줄러가 실행되지 않거나 배치가 실패한 것으로 판단                             |
| 실제 원인        | 값을 확인하던 Redis 연결 방향이 달랐음                                           |
| 구현 방향        | `ranking` Redis connection에 글로벌 누계 점수 저장                               |
| 확인 방법        | `ranking` Redis 기준으로 prefix가 붙은 실제 키 조회                              |
| 최종 결과        | Redis에 누계 점수가 정상 저장되는 것을 확인했고, 스케줄러가 동작한다는 것도 확인 |

결국 이 검증을 통해 스케줄러가 5분 주기로 배치 커맨드를 실행하고 있고, 배치 결과가 ranking Redis에 정상적으로 저장된다는 것을 확인했다.

이 과정에서 단순히 로그만 보는 것보다, 실제 결과물인 Redis 값을 직접 확인하는 것이 가장 확실하다는 것도 알 수 있었다.

이번 작업의 핵심은 글로벌 미션을 개인 랭킹과 분리해서 바라보는 것이었다.
글로벌 미션은 순위가 아니라 전체 누계 점수 하나만 필요하다.
그래서 **Redis Sorted Set 기반 랭킹 구조가 아니라 Redis String 캐시로 전체 점수를 저장**했다.
테스트 역시 커맨드 출력 문자열이 아니라 Redis 캐시 값을 직접 검증하는 방식으로 변경했다.

정리하면 다음과 같다.

```
글로벌 미션 누계 점수는 개인 랭킹과 별개다.
전체 NORMAL 플레이 점수만 합산한다.
PRACTICE 점수는 제외한다.
집계 결과는 Redis String 캐시에 저장한다.
해당 Redis 값은 ranking Redis connection 기준으로 저장된다.
스케줄러를 통해 주기적으로 재집계한다.
테스트는 Artisan::call() 후 Redis 캐시 값을 직접 검증한다.
$this->info()는 콘솔 출력이고, 로그 파일에 남기려면 Log::info()가 필요하다.
Redis CLI에서 확인할 때는 Laravel Redis prefix와 connection을 고려해야 한다.
최종적으로 ranking Redis에 prefix가 붙은 키로 값이 저장되는 것을 확인했고, 이를 통해 스케줄러가 정상 동작한다는 것을 검증했다.
```

배치 집계 방식은 실시간성은 떨어지지만, DB를 기준으로 다시 계산할 수 있기 때문에 정합성 측면에서는 안정적이다.
