# Jenkins EKS 배포가 Unstable로 끝났을 때, 디스크 용량 혹은 Docker 빌드 캐시

## 해당 글 작성 이유

최근 QA 쪽 환경에 새 버전을 배포하던 중 Jenkins 파이프라인이 `Finished: UNSTABLE`로 끝나는 상황을 겪었다. 메세지만 확인하면 "이미지가 안 올라간건가" 싶어서 로그를 확인하였는데, 해당 컨테이너에 들어가 확인했더니 원인은 전혀 달랐다. 해당 글에서는 로그 추적 및 읽는것, 무엇 때문에 의심했고, 최종적으로 어떤 문제였는 지 확인 후 어떻게 처리하였는지 트러블 슈팅 과정을 정리하였다.

## 문제 상황

배포 파이프라인은 대략 아래와 같은 흐름으로 동작한다

```
Build - Docker 이미지 빌드 & ECR Push

Update and Deploy to EKS - 배포용 mainfest 템플릿의 `${APP_VERSION}` 치환, SSH로 원격 서버(Bastion/ EKS 관리 서버) 접속 후 배포 명령 실행

Clean up - 임시 파일 정리
```

해당 건 중 `Update and Deploy to EKS` 단계에서 파이프라인이 실피했다.

### 이미지 태그 치환은 정상

```bash
+ sed s/${APP_VERSION}/20260101-42/g
+ echo 처리된 이미지 태그:
처리된 이미지 태그:
+ grep image: processed_deployment.yaml
          image: 123456789012.dkr.ecr.ap-northeast-1.amazonaws.com/example-api:qa-20260101-42
          image: redis:latest
```

`${APP_VERSION}` 변수가 실제 빌드 번호로 정상 치환, 배포용 mainfest에 이미지 태그가 정확히 나왔다. 즉 해당시점까지는 이슈가 없다.

### SSH 접속 후 실패

```
SSH: Connecting from host [a1b2c3d4e5f6]
SSH: Connecting with configuration [Docker] ...
SSH: EXEC: completed after 602,427 ms
SSH: Disconnecting configuration [Docker] ...
ERROR: Exception when publishing, exception message [Exec exit status not zero. Status [1]]
```

위에 숫자 중 `602,427ms`는 약 602초 거진 10분경이다. 배포 스크립트 내부에 `kubectl rollout status --timeout=600s` 같은 대기 로직이 있고, 새 pod가 10분 안에 ready 상태로 올라오지 못해 타임아웃으로 종료된 것이다. SSH 연결 자체는 성공했고, 원격에서 실행한 명령이 exit code1로 끝나며 Jenkins가 이를 실패로 처리했다.

## 빌드가 안되었을 때부터 확인

가장 먼저 든 생각은 "이미지 push가 실패한거라고"생각했다. 로그상 `Build` 단계는 앞서 끝났고, 새 태그가 정확히 반영되어 있었으나 이미지 자체는 정상적으로 ECR에 올라갔을 확률이 높다고 생각했다.

그래서 원격 서버에 직접 들어가 pod 상태를 확인했다

```bash
kubectl get pods -n example-ns -l app=example-api
kubectl desribe pod <pod-name> -n example-ns
kubectl logs <pod-name> -n example-ns --previous
```

파드 상태를 보다가, 문제는 쿠버네티스 스케줄링이나 앱 크래시가 아니라 다른곳에 있다고 생각했다. 배포 서버 자체의 디스크 사용량을 확인했을 때 이슈를 확인했다.

```bash
df -h
```

```
Filesystem      Size  Used  Avail  Use%  Mounted on
/dev/nvme0n1p2  937G  838G   52G   95%   /
```

해당 도커 명령을 실행하는 과정과 중간에 운영툴(관리자 페이지)의 컨테이너가 내려가는 이슈도 발생하여 다음과 같은 에러도 확인했다.

```
no space left on device
```

디스크가 95%까지 차 있었고, 새 이미지를 pull/build하거나 컨테이너를 새로 띄우는 과정에서 공간 부족으로 실패한것이라고 봤다. `EXEC completed after 602,427ms`는 결국 디스크 부족으로 인해 컨테이너 관련 작업이 지연되고 타임아웃까지 갔다고 생각했다.

## 300GB 해당하는 Docker 빌드 캐시

디스크 사용량을 항목별로 알아보기 위해 다음 명령을 사용했다

```bash
docker system df
```

```
TYPE            TOTAL   ACTIVE   SIZE      RECLAIMABLE
Images          100     10       200GB     150GB
Containers      20      15       10GB      2GB
Local Volumes   30      20       50GB      10GB
Build Cache     500     0        300GB     300GB
```

`Build Cache` 항목이 되게 컸고, `RECLAIMABLE`(정리 가능한 용량)도 꽉 찼다. 배포와 빌드를 반복하는 동안 쌓인 캐시가 디스크 공간 대부분을 잡았다.

### Docker 빌드 캐시

Docker는 이미지를 빌드할 때 속도를 높이기 위해 각 단계의 결과물을 캐시로 저장한다. 예를들어 Docker는 각 줄마다 캐시로 남긴다.

```
FROM php:8.3-fpm
RUN apt-get update && apt-get install -y nginx
COPY composer.json composer.lock ./
RUN composer install
COPY . .
```

Docker는 각 줄마다 결과를 캐시로 남긴다.

```
[1] FROM php:8.3-fpm           → 캐시 레이어 1
[2] RUN apt-get install nginx  → 캐시 레이어 2
[3] COPY composer.json ...     → 캐시 레이어 3
[4] RUN composer install       → 캐시 레이어 4
[5] COPY . .                   → 캐시 레이어 5
```
