## 라이브 DB 작업, 안전한 운영 DB 접속 작업

운영 DB에 바로 수정 쿼리를 날렸다가 데이터를 날려 먹는 이슈가 생길수도 있기에, 실무에서는 조심히 할 것, 운영 DB에 안전하게 접속하는 구조 잡기

---

### 라이브 DB 직접 수정이 위험한 이유

롤백해도 복구될 수 있으나, 위험하다.
사고나면 스냅샷으로 롤백할 수 있으나, 이때 이슈가 생ㅇ길 수 있다.

```
[정상 상태]
    │
    ▼
00:00  ─── 새벽 스냅샷 자동 생성 (인프라 정책상 하루 1회)
    │
    │      (사용자들의 결제, 랭킹, 로그 데이터가 계속 쌓임)
    │
14:00  ─── 개발자가 라이브 DB에 실수로 잘못된 UPDATE 실행 😱
    │
    ▼
[사고 발생 시점]

만약 "가장 최신 스냅샷"으로 롤백하면?
→ 00:00 시점으로 되돌아감
→ 00:00 ~ 14:00 사이의 14시간치 데이터가 전부 증발
```

최신 스냅샷 롤백은 결국 최신 안전 지점 롤백이 아니다. 스냅샷 주기와 사고 발생 시점 사이의 모든데이터(결제내역, 유저 진행 상황, 랭킹 등)이 통째로 사랒리 수 있다.
게임 서버처럼 실시간으로 데이터가 계속 쌓이는 환경에서 하루치 손실이 CS 폭탄, 신뢰도 하락으로 이어질 수 이싿.

실무에서는 라이브 DB에 쿼리 작업을 할 때, 되게 조심히 작업하며, 수정 쿼리 등의 이슈 때는 조심하는 편이다. 별도 승인 없이 개인 판단으로 처리하는 경우는 거의 없다.
