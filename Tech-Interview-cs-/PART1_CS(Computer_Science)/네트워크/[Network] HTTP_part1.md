## 1. HTTP란?

HTTP = HyperTextTransferProtocol

하이퍼텍스트를 전송하는규칙이다.
처음엔 HTML 문서만 주고받기 위해 만들어졌으나, 지금은 인터넷에서 데이터를 주고받는 거의 모든것에 사용된다.

**HTTP로 전송할 수 있는 것들**
| 종류 | 예시 |
|------|------|
| 문서 | HTML, TEXT |
| 미디어 | 이미지, 음성, 영상, 파일 |
| 데이터 | JSON, XML (API 통신) |
| 기타 | 거의 모든 형태의 데이터 |

> 유튜브 영샹을 보든, 카카오톡 메시지를 보내든, 쇼핑몰에서 주문을 하든 전부 **HTTP**가 사용될 확률이 높다.

## 2. HTTP의 역사

```
1991년  HTTP/0.9  ── GET 메서드만 존재. 정말 기초적인 버전.
  │
1996년  HTTP/1.0  ── 메서드와 헤더가 추가됨.
  │
1997년  HTTP/1.1  ── 현재까지 가장 많이 쓰이는 버전 ⭐
  │                  (이 버전을 기준으로 공부하면 됩니다!)
  │
2015년  HTTP/2    ── 속도 성능 개선
  │
현재    HTTP/3    ── TCP 대신 UDP 사용, 더 빠른 성능 목표
```

- HTTP/1.1 HTTP/2-> TCP 기반
- HTTP/3->UDP 기반

## 3. 클라이언트 서버 구조

HTTP는 요청(Request)->응답(Response)구조로 작동한다.

```
클라이언트                        서버
   │                               │
   │  ① 요청 (Request)             │
   │ ─────────────────────────────>│
   │                               │ (결과를 만들어냄)
   │  ② 응답 (Response)            │
   │ <─────────────────────────────│
```

- 클라이언트 = 요청하는 쪽(브라우저, 앱 등)
- 서버 = 요청을 받아 처리하고 응답을 돌려주는 쪽

클라이언트와 서버를 분리하면 각자의 역할에만 집중할 수 있다.

- 클라이언트 -> UI/사용성에 집중
- 서버->데이터 처리/비지니스 로직에 집중

## 4. Stateful, Stateless

서버가 클라이언트의 상태를 기억하냐, 안하냐의 차이이다.

---

**Stateful(상태유지)**
서버가 이전 대화 내용을 기억한다.

```
고객: 이 노트북 얼마인가요?
점원: 100만원입니다.               ← "노트북" 기억

고객: 2개 구매하겠습니다.
점원: 200만원입니다. 카드/현금?    ← "노트북 2개" 기억

고객: 신용카드로 구매할게요.
점원: 200만원 결제 완료!
```

**점원이 중간에 바뀌면?**

```
고객: 2개 구매하겠습니다.
점원B: ❓ 무엇을 2개 구매하시겠어요?  ← 기억 못함!
```

> ⚠️ 항상 **같은 서버**가 응답해야 합니다.  
> 그 서버에 장애가 나면 대화가 처음부터 다시 시작됩니다.

---

### Stateless (무상태)

서버가 상태를 기억하지 않는다. 클라이언트가 매번 필요한 정보를 전부 담아서 보낸다.

```
고객: 노트북 2개를 신용카드로 구매하겠습니다.  ← 모든 정보 포함
점원C: 200만원 결제 완료!                       ← 누가 응답해도 OK
```

| 구분                    | Stateful  | Stateless                |
| ----------------------- | --------- | ------------------------ |
| 서버가 상태 기억?       | ✅        | ❌                       |
| 항상 같은 서버 필요?    | ✅        | ❌                       |
| 서버 장애 대응          | ❌ 취약   | ✅ 다른 서버로 전환 가능 |
| 서버 확장 (스케일 아웃) | ❌ 어려움 | ✅ 매우 쉬움             |

> 스케일 아웃: 서버를 여러 대 늘려서 트래픽을 분산하는 방식
> Stateless에서 어느 서버가 응답해도 괜찮고 서버를 쉽게 늘릴 수 있다.

- Stateless -> 로그인 없는 단순 정보 조회 ㅍ페이지
- Stateful 필요 -> 로그인 상태 유지 (쿠키, 세션 활용)
  원칙은 상태 유지는 최소한만 사용 가능

## 5. 비연결성(Connectionless)

연결 유지, 비연결
| 구분 | 연결 유지 | 비연결 (HTTP 기본) |
|------|----------|-------------------|
| 방식 | 응답 후에도 연결 유지 | 응답 후 즉시 연결 종료 |
| 서버 자원 | 계속 소모 | 최소한으로 유지 |

HTTP는 기본적으로 비연결 모델이다.
응답이 끝나면 바로 연결을 끊어버린다.

> 1시간 동안 수천 명이 이용해도 실제로 동시에 처리하는 요청은 수십 개 이하이다.
> 페이지를 보고 읽는 시간 동안 연결을 끊어두는 것이 훨씬 효율적이다.

**한계와 해결책**
매번 새로운 연결을 맺으면 3-way handshake 시간이 낭비된다.

```
초기 HTTP (비효율)
연결 → HTML 응답 → 종료     (0.3초)
연결 → JS 응답   → 종료     (0.3초)
연결 → 이미지 응답 → 종료   (0.3초)
                    합계: 0.9초
```

-> 해결 HTTP 지속 연결(Persistent Connections)

```
연결
  ├── HTML 응답
  ├── JS 응답
  └── 이미지 응답
종료
                    합계: 0.5초 ✅
```

## 6. HTTP 메시지 구조

HTTP로 데이터를 주고 받을 때 **정해진 형식(메시지 구조)**를 따른다.

```
┌──────────────────────────────┐
│  start-line (시작 라인)       │  ← 요청이면 요청 정보, 응답이면 상태 코드
├──────────────────────────────┤
│  header (헤더)                │  ← 부가 정보들
├──────────────────────────────┤
│  (공백 라인 - CRLF)           │  ← 헤더와 바디를 구분하는 빈 줄
├──────────────────────────────┤
│  message body (메시지 바디)   │  ← 실제 전송할 데이터
└──────────────────────────────┘
```

**요청 메시지 예시**

```
GET /search?q=hello&hl=ko HTTP/1.1    ← 시작 라인 (메서드 / 경로 / 버전)
Host: www.google.com                  ← 헤더
                                      ← 공백 라인
(body 없음)
```

**응답 메시지 예시**

```
HTTP/1.1 200 OK                          ← 시작 라인 (버전 / 상태코드 / 이유)
Content-Type: text/html;charset=UTF-8   ← 헤더
Content-Length: 3423                    ← 헤더
                                        ← 공백 라인
<html>                                  ← 바디 (실제 데이터)
  <body>...</body>
</html>
```

**주요 상태 코드**

| 코드                        | 의미                   |
| --------------------------- | ---------------------- |
| `200 OK`                    | 성공                   |
| `201 Created`               | 생성 성공              |
| `400 Bad Request`           | 클라이언트 잘못된 요청 |
| `401 Unauthorized`          | 인증 필요              |
| `403 Forbidden`             | 권한 없음              |
| `404 Not Found`             | 리소스 없음            |
| `500 Internal Server Error` | 서버 내부 오류         |

**주요 헤더**
| 헤더 | 설명 |
| --------------------------- | ---------------------- |
| `Content-Type` | 전송하는 데이터 형식 |
| `Content-Length` | 데이터 크기 |
| `Host` | 요청하는 서버 주소 |
| `User-Agent` | 클라이언트(브라우저) 정보 |
| `Authorization` | 인증정보 |

## 7. API URI 설계

> 리소스(명사)만 URI에

**URI 설계**

```
GET    /members          ← 회원 목록 조회
GET    /members/{id}     ← 특정 회원 조회
POST   /members          ← 회원 등록
PUT    /members/{id}     ← 회원 전체 수정
PATCH  /members/{id}     ← 회원 부분 수정
DELETE /members/{id}     ← 회원 삭제
```

> URI에서는 복수형 단어를 권장한다. -> members

## 8. HTTP 메서드 종류

**주요메서드**
| 메서드 | 역할 | 비유 |
|--------|------|------|
| `GET` | 리소스 조회 | 도서관에서 책빌려보기 |
| `POST` | 데이터 처리/등록 | 도서관에 새 책 기증하기 |
| `PUT` | 리소스 전체 대체 | 책을 통쨰로 새 책으로 교체 |
| `PATCH` | 리소스 부분 수정 | 책의 특정 페이지만 수정 |
| `DELETE` | 리소스 삭제 | 책을 도서관에서 폐기 |

## 9. GET - 조회

서버에서 데이터를 **가져올 떄** 사용한다
서버의 데이터를 변경하지 않고 단순히 읽어오는 역할이다.

```
1. 클라이언트: GET /members/100
2. 서버: 해당 회원 정보 조회
3. 서버 응답: {"username": "young", "age: 20}
```

- 데이터는 URL 쿼리 파라미터로 전달(`/search?q=hello&h1=ko`)
- 바디에 데이터를 담아 보내는 건 지원하지 않는 서버가 많이 비권장

---

## 10. POST 등록 및 처리

**서버에 데이터를 "전달하고 처리를 요청할 때"** 사용한다.

```
① 클라이언트: POST /members  + Body: { "username": "young", "age": 20 }
② 서버: 새 ID 부여 → /members/100 생성
③ 서버 응답: 201 Created, Location: /members/100
```

POST를 쓰는 상황

```
1. 새 리소스 생성  -> 회원가입, 게시글 작성
2. 프로세스 처리 -> 결재완료 -> 배달시작 -> 배달완료
3. 애매한 경우 -> 애매하면 POST
```

---

## 11. PUT - 전체 대체

**리소스를 "통째로 교체할 때"** 사용한다.
없으면 새로 생성, 있으면 완전히 덮어 쓴다.

```
기존: { "username": "young", "age": 20 }

PUT /members/100 + Body: { "age": 50 }   ← username 빠뜨림!

결과: { "age": 50 }   ← username이 사라짐!
```

> PUT은 보내지 않은 필드까지 삭제한다. 일부만 바꾸려면 PATCH를 사용해야한다.

---

## 12. PATCH - 부분 수정

**리소스의 "일부분만 수정할 때"** 사용한다.

```
기존: { "username": "young", "age": 20}

PATCH /members/100 + Body: {"age": 50}

결과: {'username": "young", "age": 50} <- username 유지
```

---

## 13. DELETE - 삭제

**리소스를 "삭제할 때"** 사용한다.

```
DELETE /members/100 -> /members/100 리소스 삭제
```

## 14. HTTP 메서드 3가지 속성

HTTP 메서드는 안전, 멱동, 캐시가능의 속성이 있다.

**안전(Safe)**

> 호출해도 리소스가 변경되지 않는다

| 메서드 | 안전             |
| ------ | ---------------- |
| GET    | ✅ 안전          |
| HEAD   | ✅ 안전          |
| POST   | ❌ 안전하지 않음 |
| PUT    | ❌ 안전하지 않음 |
| PATCH  | ❌ 안전하지 않음 |
| DELETE | ❌ 안전하지 않음 |

> 안전은 해당 리소스 자체만 고려한다.
> GET 호출해서 서버 로그가 쌓인다면 리소스 변경이 아니므로 안전의 범위가 아니다.

---

**멱등(Idempotent)**
